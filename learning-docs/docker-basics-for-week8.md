# Docker 기초와 8주차 API 서버 실행 가이드

이 문서는 8주차 과제인 `미니 DBMS - API 서버`를 진행하면서 Docker 환경을 어떻게 이해하고 활용하면 좋은지 설명하는 학습용 문서입니다.

목표는 아래 두 가지입니다.

1. 8주차 API 서버 과제에서 Docker가 왜 유용한지 이해한다.
2. 컨테이너 안에서 `sqlapi_server`를 빌드, 실행, 테스트할 때 무엇을 봐야 하는지 이해한다.

## 한 줄 요약

Docker는 **팀원 모두가 거의 같은 Linux 실행 환경에서 API 서버를 빌드하고 테스트하게 해 주는 도구**입니다.

7주차에서는 SQL 처리기와 B+ 트리 구현을 같은 환경에서 돌리는 데 의미가 컸다면, 8주차에서는 거기에 더해 아래가 중요해집니다.

- HTTP 서버 실행
- 포트 매핑
- `schema/`, `data/` 경로 주입
- worker thread 수와 컨테이너 자원 관계

즉, 이번 주차의 Docker는 단순 빌드 도구가 아니라 **API 서버 실행 환경을 재현하는 장치**로 이해하면 됩니다.

## 왜 8주차에서 Docker가 더 중요해졌는가

7주차까지는 주로 CLI와 내부 엔진 동작이 중심이었습니다.

하지만 8주차부터는 아래 요소가 추가되었습니다.

- `HTTP/1.1 over TCP` 서버
- `GET /health`, `POST /query` 엔드포인트
- worker thread pool
- bounded task queue
- `--host`, `--port`, `--worker-count`, `--queue-capacity`
- `--schema-dir`, `--data-dir`

이 말은 곧 실행 환경이 아래와 같이 조금 더 복잡해졌다는 뜻입니다.

- 어떤 포트로 서버를 열 것인가
- 컨테이너 밖에서 그 포트에 접근할 수 있는가
- 서버 프로세스의 현재 작업 디렉터리가 어디인가
- 상대 경로 `schema`, `data`가 실제로 어디를 가리키는가
- 컨테이너가 CPU와 메모리를 얼마나 갖고 있는가

Docker는 이런 조건을 팀 전체가 비슷하게 맞추는 데 도움이 됩니다.

## 8주차 API 서버를 Docker 안에서 본다는 뜻

현재 서버는 아래처럼 동작합니다.

```text
client
  -> listen socket
  -> accept thread
  -> bounded queue
  -> worker thread
  -> HTTP parsing
  -> query handler
  -> sql engine adapter
  -> storage / index
```

Docker 안에서 본다면 이 전체가 **컨테이너라는 실행 상자 안에서 돌아가는 것**입니다.

즉, 아래 질문을 항상 같이 생각해야 합니다.

- 서버 프로세스는 컨테이너 안에서 어떤 포트에 bind하는가
- 호스트 PC에서는 어떤 포트로 접근하는가
- `schema/`, `data/` 디렉터리는 컨테이너 안 어디에 있는가
- 테스트나 `curl`은 컨테이너 안에서 하는가, 밖에서 하는가

## 컨테이너 자원과 worker-count의 관계

8주차에서 자주 헷갈리는 부분이 이것입니다.

```text
메모리가 많으면 worker_count를 크게 잡아야 하나?
```

반은 맞고, 반은 아닙니다.

현재 API 서버에서 worker 수는 메모리만 보고 정하는 값이 아닙니다.

더 중요한 기준은 아래입니다.

- 실제 CPU 개수
- 같은 테이블 요청 비율
- 파일 I/O 비율
- 락 경합 빈도
- queue가 자주 차는지 여부

왜냐하면 현재 구조는 아래 제약이 있기 때문입니다.

- 같은 테이블 요청은 `schema.storage_name` 기준 exclusive lock으로 직렬화된다.
- 전역 `TableIndexRegistry`는 mutex로 보호된다.
- 요청마다 schema 로딩, CSV 파일 접근, 인덱스 접근이 섞인다.

즉, worker를 많이 늘려도 항상 성능이 선형으로 좋아지지는 않습니다.

### 지금 프로젝트의 기본값

기본 서버 설정은 아래와 같습니다.

- `host = 127.0.0.1`
- `port = 8080`
- `worker_count = 4`
- `queue_capacity = 64`

이 값은 컨테이너 자원을 자동 계산해서 고른 값이 아니라, 문서와 구현에서 정한 **보수적인 시작값**입니다.

학습 관점에서는 아래처럼 이해하면 충분합니다.

- `4`: 안전한 기본값
- `8`: 자원이 충분하면 시도해볼 수 있는 값
- `16` 이상: 병렬 요청 패턴과 락 경쟁을 보고 판단해야 하는 값

## 포트 매핑은 왜 필요한가

컨테이너 안에서 서버가 `127.0.0.1:8080`에 떠 있어도, 그 포트가 자동으로 호스트 PC에 보이는 것은 아닙니다.

Docker에서는 보통 아래 두 층을 생각해야 합니다.

- 컨테이너 내부 포트
- 호스트 PC 외부 포트

예를 들면:

```text
호스트 18080 -> 컨테이너 8080
```

이렇게 매핑해야 호스트에서 아래처럼 접근할 수 있습니다.

```bash
curl http://127.0.0.1:18080/health
```

만약 포트 매핑을 하지 않으면, 서버는 컨테이너 안에서는 떠 있어도 호스트에서 바로 접근하기 어렵습니다.

## `schema/`, `data/` 경로를 왜 신경 써야 하는가

현재 서버는 기본적으로 아래 경로를 사용합니다.

- `--schema-dir schema`
- `--data-dir data`

그리고 상대 경로는 **서버 프로세스의 현재 working directory 기준**으로 해석됩니다.

즉, 컨테이너 안에서 서버를 어디서 실행하느냐가 중요합니다.

예를 들어 컨테이너 작업 디렉터리가 저장소 루트라면:

- `schema`는 `./schema`
- `data`는 `./data`

를 뜻하게 됩니다.

이 때문에 Docker를 쓸 때는 아래 두 가지를 항상 확인해야 합니다.

1. 프로젝트 폴더가 컨테이너 안 어디에 마운트되어 있는가
2. 서버를 어떤 작업 디렉터리에서 실행하는가

이 두 가지가 어긋나면 서버가 시작하면서 아래 같은 오류를 낼 수 있습니다.

- `--schema-dir must point to an existing directory`
- `--data-dir must point to an existing directory`

## Docker 안에서 week8 서버를 다룰 때의 기본 흐름

학습용으로는 아래 순서로 이해하면 충분합니다.

1. 컨테이너 안에 프로젝트 폴더를 마운트한다.
2. 작업 디렉터리를 프로젝트 루트로 맞춘다.
3. `make all`로 빌드한다.
4. `./build/bin/sqlapi_server ...`로 서버를 실행한다.
5. 다른 터미널에서 `curl` 또는 테스트 러너로 확인한다.

흐름을 그림으로 보면 아래와 같습니다.

```text
호스트 프로젝트 폴더
   |
   +-- bind mount
   |
   v
컨테이너 작업 디렉터리
   |
   +-- make all
   +-- ./build/bin/sqlapi_server
   +-- ./build/bin/test_runner
   |
   v
컨테이너 내부의 schema/, data/, build/bin/
```

## 자주 쓰는 명령을 어떻게 이해하면 좋은가

여기서는 실제 팀의 Dockerfile이나 compose 파일을 설명하기보다, 개념적으로 어떤 명령이 왜 필요한지 이해하는 데 집중합니다.

### 1. 빌드

```bash
make all
```

이 명령은 컨테이너 안에서 아래 바이너리를 만듭니다.

- `build/bin/sqlparser`
- `build/bin/sqlapi_server`
- `build/bin/test_runner`

### 2. API 서버 실행

```bash
./build/bin/sqlapi_server \
  --host 127.0.0.1 \
  --port 8080 \
  --worker-count 4 \
  --queue-capacity 64 \
  --schema-dir schema \
  --data-dir data
```

이 명령을 Docker 안에서 볼 때는 아래 의미로 읽으면 됩니다.

- `--host 127.0.0.1`: 컨테이너 내부 loopback에 bind
- `--port 8080`: 컨테이너 내부 8080 포트 사용
- `--worker-count 4`: worker thread 수를 4개로 시작
- `--schema-dir schema`, `--data-dir data`: 현재 작업 디렉터리 기준 상대 경로 사용

### 3. 테스트 실행

```bash
make test
```

현재 테스트는 엔진 회귀, CLI, API 통합 테스트를 하나로 묶어 실행합니다.

즉, Docker 안에서 `make test`가 성공한다는 것은:

- 최소한 같은 Linux 계열 환경에서
- 현재 소스 기준 빌드와 자동 테스트가 재현된다는 뜻입니다.

## 컨테이너 자원을 확인하는 방법

Docker 안에서 worker 수나 성능을 논의할 때는 아래 정보를 먼저 보는 습관이 좋습니다.

### CPU 개수

```bash
nproc
```

또는

```bash
getconf _NPROCESSORS_ONLN
```

### 메모리 정보

```bash
grep -E 'MemTotal|MemAvailable' /proc/meminfo
```

이런 명령은 아래 질문에 답할 때 유용합니다.

- 이 컨테이너는 병렬 worker를 몇 개까지 감당할 여력이 있는가
- 기본값 `4`가 너무 작은가
- `8`이나 `16`을 시도해볼 근거가 있는가

## 성능과 worker 수를 볼 때 주의할 점

Docker 안에서 자원이 충분해 보여도, worker를 무조건 많이 늘리는 것이 정답은 아닙니다.

현재 구조상 아래 상황에서는 worker를 늘려도 이득이 작을 수 있습니다.

- 대부분의 요청이 같은 테이블에 몰린다
- 파일 I/O가 병목이다
- 락 경쟁이 심하다
- queue는 비지 않지만 응답 시간이 줄지 않는다

따라서 worker 수는 아래처럼 조정하는 것이 좋습니다.

1. 기본값 `4`로 시작한다.
2. `8`로 올려 본다.
3. 필요하면 `16`까지 비교한다.
4. 평균 응답 시간, queue full 여부, CPU 사용률, 같은 테이블 요청 비율을 함께 본다.

즉, 메모리만 보고 판단하지 말고 **CPU + 락 구조 + I/O 패턴**을 같이 봐야 합니다.

## 초심자가 특히 헷갈리기 쉬운 포인트

### 1. 컨테이너 안의 `127.0.0.1`과 호스트의 `127.0.0.1`은 다르다

컨테이너 안에서 서버가 `127.0.0.1:8080`에 떠 있다고 해서, 호스트에서도 자동으로 같은 주소로 바로 접근되는 것은 아닙니다.

포트 매핑이 필요할 수 있습니다.

### 2. 상대 경로는 컨테이너의 현재 작업 디렉터리 기준이다

`schema`, `data`는 내 에디터가 보는 경로가 아니라, **서버 프로세스가 실행되는 위치 기준**으로 해석됩니다.

### 3. Docker는 환경을 맞춰 주는 도구이지 성능을 자동으로 최적화해 주는 도구는 아니다

Docker를 쓴다고 worker 수, 락 전략, I/O 병목이 자동으로 해결되지는 않습니다.

### 4. `make test` 성공과 성능 최적화는 다른 문제다

테스트 통과는 기능 검증이고, worker 수 최적화는 성능 튜닝입니다. 둘은 구분해서 봐야 합니다.

## 7주차 문서와 비교해 기억하면 좋은 점

7주차 Docker 문서가 "같은 C 개발 환경과 성능 측정 기반을 맞춘다"에 더 초점이 있었다면, 8주차에서는 아래가 추가되었습니다.

- API 서버 프로세스 실행
- 포트와 네트워크 접근
- worker thread 수 조정
- `schema/`, `data/` 경로 주입
- HTTP 요청 검증과 API 테스트 재현

즉, 이번 주차의 Docker는 **빌드 환경 + 서버 실행 환경 + 테스트 재현 환경**을 동시에 설명해야 합니다.

## 기억하면 좋은 문장

아래 문장만 기억해도 이번 주차 Docker 감각을 잡는 데 충분합니다.

- Docker는 8주차 API 서버를 같은 Linux 환경에서 빌드하고 실행하게 해 주는 도구다.
- 포트 매핑이 없으면 컨테이너 안 서버가 떠 있어도 호스트에서 바로 접근하기 어려울 수 있다.
- `schema/`, `data` 상대 경로는 컨테이너의 현재 작업 디렉터리 기준으로 해석된다.
- `worker-count`는 메모리만 보고 정하지 않고, CPU 개수와 락 경쟁, I/O 패턴을 함께 본다.
- Docker는 환경 재현성을 높여 주지만, 락 병목이나 코드 버그를 대신 해결해 주지는 않는다.
