# 8주차 API 서버 코드 리뷰 가이드

## 1. 문서 목적

이 문서는 8주차에 추가된 API 서버 코드를 리뷰하면서,
"어디부터 어떤 순서로 봐야 전체 구조가 이해되는가"를 초심자 기준으로 안내하기 위해 작성했다.

이번 8주차 구현의 핵심은 기존 7주차 SQL 엔진을 새로 갈아엎는 것이 아니라,
그 바깥에 API 서버 계층을 얹고 병렬 요청 처리 구조를 추가한 것이다.

따라서 코드 리뷰도 "기존 엔진 전체를 처음부터 다시 읽는 방식"보다
"8주차에서 추가된 바깥 계층을 먼저 읽고, 어디서 기존 엔진으로 내려가는지 확인하는 방식"이 훨씬 효율적이다.

이 문서는 다음 질문에 답하도록 구성한다.

- 8주차에서 새로 추가된 코드는 어디인가?
- 어떤 파일부터 읽어야 전체 흐름이 잡히는가?
- 각 단계에서 무엇을 확인해야 하는가?
- 병렬 처리와 동시성 보호는 어디서 확인해야 하는가?
- 기존 SQL 엔진과의 연결은 어디서 일어나는가?
- 테스트 코드는 어떤 관점으로 읽어야 하는가?
- 발표나 Q&A에서 자주 나오는 질문에 어떻게 대비할 수 있는가?

## 2. 이 문서를 읽는 방법

이 문서는 "요약 설명"이 아니라 "코드 리뷰 안내서"다.
즉 이 문서만 읽고 끝내는 것이 아니라, 문서의 순서에 따라 실제 파일을 함께 열어 보는 것이 목적이다.

권장 방식은 아래와 같다.

1. 먼저 8주차 문서 3개를 읽는다.
2. 그다음 서버 진입점부터 한 파일씩 따라간다.
3. 요청 1건이 어디를 거쳐 어디서 멈추는지 머릿속으로 그린다.
4. 병렬 처리, 락, queue full 같은 예외 시나리오를 따로 확인한다.
5. 마지막으로 테스트 코드를 보며 "문서와 구현이 실제로 맞물리는가"를 검증한다.

이 과정을 거치면 "파일은 많은데 무엇이 중요한지 모르겠다"는 상태에서 벗어나기 쉽다.

## 3. 먼저 읽을 문서

코드를 보기 전에 아래 문서를 먼저 읽는 것을 권장한다.

1. `docs/week8-architecture.md`
2. `docs/week8-api-spec.md`
3. `docs/week8-requirements.md`
4. `README.md`

이 순서를 추천하는 이유는 다음과 같다.

- 아키텍처 문서는 계층 책임과 실행 흐름을 먼저 잡아 준다.
- API 명세서는 외부 계약과 오류 처리 기준을 알려 준다.
- 요구사항 문서는 이번 주차에서 무엇이 본질인지 경계를 잡아 준다.
- README는 실제 구현 요약과 발표 관점 설명을 제공한다.

코드 리뷰 중 길을 잃기 쉬운 이유는 네트워크 코드, HTTP 코드, SQL 코드, 스토리지 코드가 한 저장소에 모두 들어 있기 때문이다.
문서를 먼저 읽으면 어떤 계층이 8주차 추가분이고 어떤 계층이 기존 재사용분인지 훨씬 빨리 구분할 수 있다.

## 4. 이번 주차를 한 문장으로 요약하면

이번 8주차는 다음 한 문장으로 요약할 수 있다.

`브라우저나 외부 클라이언트가 보낸 HTTP 요청을, 스레드 풀과 작업 큐를 거쳐 기존 SQL 엔진 실행으로 연결한 주차다.`

이 문장을 조금 더 풀면 다음과 같다.

- 클라이언트는 더 이상 CLI만 쓰지 않고 HTTP로 요청을 보낸다.
- 서버는 요청을 직접 처리하지 않고 worker thread에게 분배한다.
- worker는 HTTP 요청을 해석한 뒤 SQL 실행을 서비스 계층으로 넘긴다.
- 엔진 어댑터가 기존 SQL 처리기와 B+ 트리 인덱스 계층을 호출한다.
- 같은 테이블은 락으로 직렬화하고, 다른 테이블은 병렬 처리한다.

이 큰 그림을 먼저 잡고 나면, 각 파일이 어디에 속하는지 이해하기 쉬워진다.

## 5. 코드 리뷰 전에 머릿속에 넣어둘 큰 그림

이번 구현의 계층 구조는 아래와 같이 볼 수 있다.

```text
client
  -> server
  -> http
  -> api
  -> service
  -> engine adapter
  -> sql / execution
  -> storage / index
```

이 구조를 코드 리뷰 관점에서 해석하면 다음과 같다.

- `client`: 브라우저, curl, 테스트 코드처럼 바깥에서 요청을 보내는 쪽
- `server`: 소켓, accept loop, queue, worker thread pool을 관리하는 쪽
- `http`: 바이트 스트림을 HTTP 의미 구조로 해석하는 쪽
- `api`: 엔드포인트 단위로 요청을 받는 쪽
- `service`: API 계층과 엔진 계층 사이를 느슨하게 잇는 완충 지점
- `engine adapter`: 기존 SQL 엔진을 HTTP/API 세계에 맞게 감싸는 연결부
- `sql / execution / storage / index`: 7주차 재사용 영역

초심자가 흔히 헷갈리는 부분은 "어디서 SQL이 실제로 실행되는가"이다.
정답은 `api`도 아니고 `service`도 아니며,
실제 엔진 진입은 `engine adapter`에서 일어난다.

## 6. 8주차에서 새로 추가된 코드 범위

이번 8주차에서 중심적으로 봐야 하는 코드는 아래 디렉터리다.

- `src/app/sqlapi_server_main.c`
- `src/server/`
- `src/http/`
- `src/api/`
- `src/service/`
- `src/engine/`

반대로 아래 디렉터리는 "기존 엔진 재사용 영역"으로 보면 된다.

- `src/sql/`
- `src/execution/`
- `src/storage/`
- `src/index/`

즉 리뷰의 기본 전략은 이렇다.

1. 8주차 추가 계층을 먼저 읽는다.
2. 각 계층이 어디서 기존 엔진으로 내려가는지 확인한다.
3. 그 뒤 필요한 기존 엔진 파일만 목적에 맞게 찾아 읽는다.

이 순서를 지키지 않으면 `lexer.c`, `parser.c`, `executor.c`, `storage.c`, `bptree.c` 같은 파일을 먼저 읽다가,
이번 주차에서 새로 붙인 서버 구조를 놓치기 쉽다.

## 7. 요청 1건을 예시로 먼저 따라가 보기

코드 리뷰를 시작하기 전에, 요청 한 건이 어떻게 흐르는지를 먼저 머릿속으로 그려 두는 것이 좋다.

예를 들어 아래 요청을 생각해 보자.

```http
POST /query HTTP/1.1
Host: 127.0.0.1:8080
Content-Type: application/json
Content-Length: 44

{"sql":"SELECT * FROM users WHERE id = 1;"}
```

이 요청은 대략 아래 순서로 흐른다.

1. 서버가 TCP 연결을 accept한다.
2. accept thread가 이 연결을 task queue에 넣는다.
3. worker thread 하나가 queue에서 연결을 꺼낸다.
4. worker가 HTTP request를 읽고 파싱한다.
5. router가 `POST /query`를 `query_handler`로 보낸다.
6. `query_handler`가 JSON에서 `sql` 문자열을 꺼낸다.
7. `db_service`가 엔진 어댑터에 실행을 위임한다.
8. 엔진 어댑터가 lexer, parser, executor를 호출한다.
9. 필요하면 table lock을 잡는다.
10. 실행 결과를 캡처해 JSON 응답으로 바꾼다.
11. worker가 응답을 보내고 소켓을 닫는다.

이 흐름은 코드 리뷰 전체의 기준선이다.
각 파일을 읽을 때마다 "지금 이 파일은 위 11단계 중 몇 번째에 해당하는가?"를 자문하면 길을 잃기 어렵다.

## 8. 추천 읽기 순서

### 8.1 1단계: 서버 진입점부터 본다

가장 먼저 볼 파일:

- `src/app/sqlapi_server_main.c`

이 파일에서 확인할 것:

- 서버 실행 옵션은 무엇인가
- 기본값은 무엇인가
- 어떤 설정을 `SqlApiServerConfig`에 넣는가
- 서버 생성, 시작, 대기 흐름이 어떻게 이어지는가
- 옵션 검증 실패가 어디서 드러나는가

이 파일은 "프로세스가 어떻게 시작되는가"를 보여 준다.
복잡한 비즈니스 로직은 없지만, 전체 시스템의 바깥 경계를 잡는 데 가장 적합하다.

리뷰 질문:

- 서버를 어떤 경로로 실행하는가?
- `--host`, `--port`, `--worker-count`, `--queue-capacity`, `--schema-dir`, `--data-dir`는 어디서 처리되는가?
- 잘못된 옵션은 어디서 실패하는가?
- 이 파일이 너무 많은 일을 하고 있는가, 아니면 단순한 진입점 역할만 하는가?

초심자 팁:

이 파일은 "main 함수"이기 때문에 단순해 보여도 중요하다.
프로그램의 바깥 경계를 보여 주고, 어떤 설정이 아래 계층으로 흘러가는지 확인할 수 있기 때문이다.

### 8.2 2단계: 서버 런타임 구조를 본다

다음으로 볼 파일:

- `src/server/server.c`
- `src/server/task_queue.c`
- `src/server/worker_pool.c`

이 단계에서 확인할 것:

- listen socket 생성
- accept loop
- bounded queue
- worker thread pool
- queue full 시 `503` 반환
- shutdown 처리

이 영역이 8주차 핵심 중 하나다.
공지사항의 `스레드 풀`, `병렬 SQL 처리`, `멀티 스레드 동시성 이슈`에 가장 직접적으로 대응한다.

리뷰 질문:

- 요청 하나가 어떤 단위로 queue에 들어가는가?
- worker는 queue에서 무엇을 꺼내는가?
- worker 수는 고정인가?
- queue가 가득 차면 어떤 스레드가 무엇을 하는가?
- shutdown 시 accept thread, worker thread, queue는 어떤 순서로 정리되는가?

리뷰 포인트:

- `server.c`는 구조를 조립하는 중심 파일이다.
- `task_queue.c`는 생산자/소비자 동기화 구현을 확인하는 파일이다.
- `worker_pool.c`는 스레드 생성과 조인의 보조 헬퍼다.

초심자 관점에서 특히 봐야 할 점:

- queue에 들어가는 것은 "요청 문자열"이 아니라 "이미 accept된 client socket"이다.
- 따라서 queue에 대기 중인 작업도 소켓 자원을 사용한다.
- queue가 가득 찬 경우는 "조용히 버림"이 아니라 `503 QUEUE_FULL`로 명시적으로 거절한다.

### 8.3 3단계: HTTP 파싱과 라우팅을 본다

다음으로 볼 파일:

- `src/http/http_request.c`
- `src/http/http_response.c`
- `src/http/router.c`

이 단계에서 확인할 것:

- request line, header, body 파싱
- `Content-Length`, `Content-Type` 검증
- request line과 header 크기 제한
- 상태 코드와 JSON 오류 응답 조립
- 경로와 메서드 라우팅

이 계층은 네트워크 바이트를 "HTTP 의미 구조"로 바꾸는 역할이다.
SQL 실행 자체는 하지 않지만, 실제 오류 응답의 상당수가 여기서 결정된다.

리뷰 질문:

- 잘못된 헤더는 어디서 걸러지는가?
- `Transfer-Encoding: chunked`는 어디서 막는가?
- `404`, `405`, `431`, `411`, `413`, `501`은 주로 어디서 나오는가?
- router는 어느 시점에 API handler로 넘기는가?

리뷰 포인트:

- 이 계층의 책임은 "프로토콜 검증"이다.
- SQL 에러와 HTTP 형식 에러를 구분해서 읽는 것이 중요하다.
- HTTP 계층이 SQL parser를 직접 부르면 계층이 섞인 것이다.

초심자 팁:

이 부분은 생소한 C 문자열 처리와 소켓 읽기 코드가 많아서 지치기 쉽다.
모든 줄을 세세히 읽기보다, "무슨 입력을 받아 어떤 구조체로 만드는가"를 먼저 보는 것이 좋다.

### 8.4 4단계: API 엔드포인트 처리 코드를 본다

다음으로 볼 파일:

- `src/api/root_handler.c`
- `src/api/health_handler.c`
- `src/api/query_handler.c`

이 단계에서 확인할 것:

- `GET /`
- `GET /health`
- `POST /query`
- JSON body 파싱
- 요청 필드 검증
- 서비스 호출 전후 처리

리뷰 질문:

- 브라우저 진입 페이지는 어디서 HTML을 생성하는가?
- `GET /health` 응답 필드는 어디서 채우는가?
- `POST /query`에서 `sql` 필드는 어디서 추출하는가?
- JSON 오류와 SQL 오류는 어디서 갈라지는가?

리뷰 포인트:

- `root_handler.c`는 브라우저 시연 화면을 제공한다.
- `health_handler.c`는 서버 상태 확인용 경량 엔드포인트다.
- `query_handler.c`는 실제 SQL 실행 요청의 API 진입점이다.

초심자 관점에서 중요한 점:

- `root_handler.c`는 "프론트엔드 앱"이 아니라 얇은 HTML 페이지다.
- 이 페이지도 결국 내부적으로는 `POST /query`를 호출하는 클라이언트일 뿐이다.
- 따라서 진짜 핵심 API는 여전히 `POST /query`다.

### 8.5 5단계: 서비스 계층을 본다

다음으로 볼 파일:

- `src/service/db_service.c`

이 파일은 얇지만 중요하다.
API 계층이 엔진 세부 구현에 직접 붙지 않도록 완충 역할을 한다.

리뷰 질문:

- API 핸들러는 왜 바로 엔진 어댑터를 호출하지 않는가?
- 지금은 얇지만 향후 정책이 들어갈 지점은 어디인가?
- 이 계층이 있어야 테스트와 책임 분리가 쉬워지는가?

리뷰 포인트:

- 현재 로직이 단순하더라도 계층 분리 의도를 읽는 것이 중요하다.
- 발표나 Q&A에서는 "왜 service 계층을 둔 것인지" 설명할 수 있어야 한다.

초심자 팁:

`service`가 너무 얇아 보이면 "없애도 되지 않나?"라는 생각이 들 수 있다.
그 자체는 자연스러운 의문이다.
다만 이 프로젝트에서는 API와 엔진 사이에 완충층을 두어,
핸들러가 엔진 내부 세부사항에 직접 달라붙지 않도록 한 것으로 이해하면 된다.

### 8.6 6단계: 엔진 연결부를 본다

다음으로 볼 파일:

- `src/engine/sql_engine_adapter.c`
- `src/engine/engine_lock_manager.c`

이 단계가 리뷰의 핵심이다.
API 서버 세계와 기존 SQL 엔진 세계가 여기서 만난다.

이 단계에서 확인할 것:

- SQL 길이 검증
- lexer와 parser 호출
- AST에서 대상 테이블 이름 추출
- schema 로드
- `schema.storage_name` 확정
- 테이블 락 획득과 해제
- executor 호출
- 출력 캡처
- 오류 코드 분류

리뷰 질문:

- 왜 raw table name이 아니라 `storage_name`을 락 키로 쓰는가?
- 락은 SQL 실행 전 어느 시점에 잡히는가?
- 오류 매핑은 어디서 일어나는가?
- `SELECT`와 `INSERT` 결과는 어떻게 API 응답으로 바뀌는가?
- 이 파일이 너무 많은 책임을 지고 있지는 않은가?

리뷰 포인트:

- `sql_engine_adapter.c`는 8주차 구현의 실제 중심 파일이다.
- `engine_lock_manager.c`는 테이블별 직렬화 정책을 확인하는 파일이다.

초심자 관점에서 중요한 점:

- 이 프로젝트의 lock 기준은 "SQL 문자열에 보이는 이름"이 아니다.
- `load_schema()` 이후 확정되는 `schema.storage_name`이 기준이다.
- 따라서 alias처럼 보이는 이름이 달라도 같은 물리 테이블이면 같은 락으로 묶인다.

## 9. 왜 `engine adapter`가 중요한가

코드 리뷰를 하다 보면 `server.c`가 눈에 띄고, `root_handler.c`가 눈에 띄고, `executor.c`도 눈에 띈다.
하지만 이번 주차를 가장 잘 설명해 주는 파일은 결국 `sql_engine_adapter.c`다.

그 이유는 다음과 같다.

- 8주차의 새 요구사항과 7주차의 기존 엔진을 실제로 이어 주기 때문이다.
- 락, 실행, 출력 캡처, 오류 변환이 모두 여기서 만난다.
- API 서버가 storage나 index를 직접 건드리지 않고도 기존 엔진을 재사용하게 해 주기 때문이다.

즉 발표나 리뷰에서 "이번 주차의 본질이 무엇인가?"를 설명할 때,
이 어댑터 계층을 빼놓고 설명하기 어렵다.

## 11. 병렬 처리 구조를 리뷰할 때 꼭 확인할 것

이번 주차의 중요한 차별점은 "HTTP 서버를 붙였다"보다도
"병렬 요청 처리를 어디까지 안전하게 허용했는가"에 있다.

병렬 처리 구조를 읽을 때는 아래 네 가지를 구분해야 한다.

### 11.1 accept loop

- 새 연결을 받는 역할
- 요청 내용을 해석하지는 않음
- queue에 넣을지, queue가 가득 찼으면 거절할지 결정

### 11.2 bounded queue

- 이미 accept된 작업을 잠시 대기시키는 공간
- 무한정 쌓이지 않도록 용량 제한이 있음
- 가득 차면 `503 QUEUE_FULL`

### 11.3 worker thread pool

- queue에서 작업을 꺼내 실제 HTTP 처리와 SQL 실행을 수행
- worker 개수는 고정
- 매 요청마다 새 스레드를 만들지 않음

### 11.4 table-level exclusive lock

- 같은 테이블 작업은 동시에 실행하지 않도록 막음
- 다른 테이블은 병렬 실행 가능
- 현재는 read/write를 구분하지 않는 단순한 배타 락

이 네 개를 한 번에 이해해야,
"worker는 병렬인데 왜 어떤 요청은 기다리나?" 같은 질문에 답할 수 있다.

## 12. 현재 구조의 장점과 한계

코드 리뷰는 단순히 "무엇을 했다"를 확인하는 것뿐 아니라,
"왜 이렇게 했는가"와 "어디까지가 한계인가"를 읽는 과정이기도 하다.

### 12.1 장점

- 구조가 단순해 구현과 설명이 쉽다.
- 기존 엔진 재사용 경계가 비교적 명확하다.
- 병렬 처리 요구사항과 최소 동시성 보호 요구를 충족한다.
- 테스트로 queue full, 동시성, shutdown, 복구를 확인할 수 있다.

### 12.2 한계

- lock-aware scheduling이 아니다.
- worker가 작업을 먼저 가져간 뒤 락에서 대기할 수 있다.
- 같은 테이블 요청이 몰리면 여러 worker가 함께 묶일 수 있다.
- keep-alive, chunked transfer, transaction, authentication 같은 실서비스 기능은 범위 밖이다.

초심자가 이 한계를 읽을 때 중요한 점은 "잘못 구현했다"가 아니라
"과제 범위에 맞는 최소 구현으로 어디까지 선택했는가"를 구분하는 것이다.

## 13. 코드 리뷰 중 자주 놓치는 핵심 포인트

### 13.1 브라우저별 인덱스가 아니라 서버 프로세스 공유 인덱스다

브라우저는 단지 클라이언트일 뿐이다.
테이블 인덱스는 서버 프로세스 안의 전역 레지스트리에 올라가므로,
한 브라우저가 준비한 인덱스를 다른 브라우저 요청도 재사용할 수 있다.

### 13.2 같은 테이블 요청은 무조건 직렬화된다

현재 정책은 read/write 분리 락이 아니라 `table-level exclusive lock`이다.
따라서 같은 테이블에 대한 `SELECT`와 `INSERT`도 서로 기다릴 수 있다.

### 13.3 빠른 조회는 `WHERE id = ...` 경로다

`name`, `age` 같은 일반 컬럼 조건은 기본적으로 CSV 스캔 경로를 탄다.
반면 `WHERE id = <integer>`는 내부 숨은 PK 기준 인덱스 경로를 탄다.

### 13.4 첫 `id` 조회는 느릴 수 있다

인덱스가 아직 메모리에 없으면 첫 접근 시 CSV 전체를 읽어 lazy rebuild가 일어날 수 있다.
따라서 첫 `id` 조회가 항상 즉시 빠른 것은 아니다.

### 13.5 `GET /` 프론트는 얇은 클라이언트일 뿐이다

루트 페이지는 별도 실행 계약이 아니라,
결국 기존 `POST /query`를 호출하는 브라우저용 클라이언트다.

### 13.6 queue에 있는 작업도 이미 소켓을 잡고 있다

accept가 끝난 뒤 queue에 들어가므로,
queue 대기 작업도 "아직 연결 전 요청"이 아니라 이미 할당된 client socket을 들고 있다.

### 13.7 테스트는 임시 디렉터리에 실제 테이블 파일을 만든다

API 테스트는 `build/tests/api_tmp_*` 아래에 `users.meta`, `users.csv` 같은 파일을 만들어
격리된 테스트용 데이터셋으로 서버를 띄운다.

## 14. 테스트 코드를 어떻게 읽을 것인가

이번 주차는 문서와 구현뿐 아니라 테스트 코드도 중요한 학습 자료다.
테스트를 보면 "팀이 무엇을 위험하다고 봤는지"가 드러난다.

중심 파일:

- `tests/test_api_server.c`
- `tests/test_runner.c`

### 14.1 먼저 볼 포인트

- 서버 시작 옵션 검증
- `GET /`, `GET /health`, `POST /query` 정상 동작
- 잘못된 JSON, 헤더, 본문 크기 초과
- queue full
- 같은 테이블 직렬화
- 다른 테이블 병렬 처리
- alias lock 정규화
- shutdown
- restart recovery

### 14.2 테스트 코드를 읽을 때의 질문

- 문서에서 정의한 오류 코드가 실제로 이렇게 나오나?
- queue full은 어떤 방식으로 재현했나?
- 같은 테이블 직렬화는 시간 차이로 어떻게 검증했나?
- 테스트가 전역 `schema/`, `data/`를 건드리지 않고 격리되나?
- 임시 파일과 서버 정리는 어떤 방식으로 이뤄지나?

### 14.3 테스트가 주는 학습 효과

테스트는 단순한 통과/실패 도구가 아니라,
아키텍처 문서에 적힌 정책이 실제 코드에서 어떻게 구현됐는지를 보여 준다.
특히 `queue full`, `concurrency`, `alias lock` 같은 항목은,
문서만 읽을 때보다 테스트를 볼 때 훨씬 선명하게 이해된다.

## 19. 마무리

8주차 코드 리뷰에서 가장 중요한 태도는 "모든 파일을 같은 깊이로 읽지 않는 것"이다.

이번 구현의 본질은 아래 네 줄로 정리된다.

- API 서버를 기존 엔진 바깥에 추가했다.
- worker thread pool과 task queue로 병렬 요청 구조를 만들었다.
- engine adapter로 기존 SQL 처리기와 B+ 트리 인덱스를 재사용했다.
- 같은 테이블 요청은 락으로 직렬화해 최소한의 정합성을 지켰다.

따라서 리뷰도 이 본질을 중심으로 진행해야 한다.
먼저 8주차 추가 계층을 읽고, 그다음 필요한 기존 엔진 파일만 목적별로 내려가서 읽으면 훨씬 빠르게 전체 구조를 이해할 수 있다.
