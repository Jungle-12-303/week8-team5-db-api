# Docker 학습 가이드

이 문서는 7주차 과제를 진행하면서 자주 듣게 되는 `Docker`, `이미지`, `컨테이너`가 무엇인지 이해하기 위한 학습용 문서입니다.

목표는 아래 두 가지입니다.

1. Docker가 무엇인지 큰 흐름으로 이해한다.
2. 현재 SQL 처리기 + B+ 트리 과제에서 왜 Docker가 필요한지 이해한다.

## 한 줄 요약

Docker는 **프로그램이 실행되는 환경까지 함께 묶어서, 누구나 거의 같은 환경에서 실행할 수 있게 해 주는 도구**입니다.

즉, 코드만 공유하는 것이 아니라

- 운영체제 환경
- 필요한 패키지
- 컴파일러
- 실행 명령

까지 함께 고정해서 공유하는 방식입니다.

## 왜 이런 도구가 필요한가

같은 C 프로젝트라도 사람마다 개발 환경이 다를 수 있습니다.

예를 들어 아래처럼 차이가 날 수 있습니다.

- 어떤 사람은 Windows에서 작업한다.
- 어떤 사람은 macOS를 쓴다.
- 어떤 사람은 WSL이나 Linux를 쓴다.
- GCC 버전이 다르다.
- `make`가 있는 사람도 있고 없는 사람도 있다.
- 라이브러리 설치 상태가 다르다.

이때 같은 코드를 받아도 아래 같은 문제가 생깁니다.

- 내 컴퓨터에서는 빌드되는데 팀원 컴퓨터에서는 안 된다.
- 경로가 달라 실행 명령이 달라진다.
- 성능 테스트 결과가 환경 차이 때문에 흔들린다.
- 발표 직전 데모 환경에서 예상과 다르게 동작한다.

Docker는 이런 문제를 줄이기 위해 사용합니다.

## Docker를 이루는 핵심 개념

### 1. 이미지(Image)

이미지는 실행 환경의 설계도 또는 스냅샷이라고 보면 됩니다.

예를 들어 아래 정보가 이미지에 들어갈 수 있습니다.

- Ubuntu 기반 환경 사용
- `gcc`, `make`, `gdb` 설치
- 프로젝트 소스 복사
- 빌드 명령 실행
- 기본 실행 명령 설정

쉽게 말해 이미지는 "이 프로젝트를 어떤 환경에서 어떻게 실행할지 적어 둔 결과물"입니다.

### 2. 컨테이너(Container)

컨테이너는 이미지를 실제로 실행한 상태입니다.

비유하면 아래와 같습니다.

- 이미지: 붕어빵 틀
- 컨테이너: 실제로 구워진 붕어빵

즉, 이미지는 설계도이고, 컨테이너는 그 설계도로 실제 동작 중인 실행 환경입니다.

### 3. Dockerfile

Dockerfile은 이미지를 만드는 방법을 적어 둔 텍스트 파일입니다.

예를 들면 아래와 비슷한 내용이 들어갑니다.

- 어떤 베이스 OS를 사용할지
- 어떤 패키지를 설치할지
- 어떤 파일을 복사할지
- 어떤 명령으로 빌드할지
- 실행 시작 시 무엇을 할지

즉, Dockerfile은 "이 환경을 이렇게 만들어라"라고 적어 둔 레시피입니다.

### 4. 볼륨(Volume) 또는 바인드 마운트(Bind Mount)

컨테이너는 보통 일회성 실행 환경입니다.

그런데 우리는 코드 파일, 테스트 결과, 실행 로그를 컨테이너 밖에서도 유지하고 싶습니다.

이때 호스트 컴퓨터의 폴더를 컨테이너 안에 연결해서 사용합니다.

현재 프로젝트에서는 보통 아래 개념으로 이해하면 충분합니다.

- 내 PC의 프로젝트 폴더
- 컨테이너 안의 작업 폴더

이 둘을 연결해 두면, 내가 에디터에서 수정한 코드가 컨테이너 안에서도 바로 보입니다.

## 가상머신과는 무엇이 다른가

Docker는 종종 "가상환경"이라고 부르지만, 전통적인 가상머신과는 느낌이 조금 다릅니다.

가상머신은 운영체제 전체를 하나 더 띄우는 방식에 가깝고, Docker는 필요한 실행 환경만 가볍게 격리하는 방식에 가깝습니다.

학습 단계에서는 아래 정도로 이해하면 충분합니다.

- 가상머신: 컴퓨터를 하나 더 띄우는 느낌
- Docker: 프로그램 실행 상자를 따로 만드는 느낌

그래서 보통 Docker 쪽이 더 가볍고, 개발용 반복 실행에도 편합니다.

## 현재 프로젝트에서 Docker가 왜 필요한가

현재 프로젝트는 7주차 과제로 아래 요구를 해결해야 합니다.

- 기존 SQL 처리기를 유지한 채 확장해야 한다.
- C 언어로 B+ 트리 인덱스를 구현해야 한다.
- 대량 Insert와 성능 테스트를 해야 한다.
- 팀원들이 같은 결과를 재현할 수 있어야 한다.

이 상황에서 Docker가 필요한 이유는 아래와 같습니다.

### 1. 팀원 모두가 같은 C 빌드 환경을 쓰기 위해

이번 과제는 C 프로젝트입니다.

즉, 아래 도구들이 중요합니다.

- `gcc` 또는 `clang`
- `make`
- 디버깅 도구
- 테스트 실행 환경

Docker를 쓰면 팀원 각자의 PC가 Windows인지 macOS인지와 관계없이, 컨테이너 안에서는 같은 Linux 기반 도구 체인을 사용할 수 있습니다.

이렇게 되면

- 컴파일러 버전 차이
- 경로 차이
- 빌드 스크립트 차이

를 크게 줄일 수 있습니다.

### 2. 성능 테스트를 좀 더 일관되게 하기 위해

7주차 과제는 대량 데이터 Insert와 조회 성능 비교가 중요합니다.

예를 들면 아래를 비교하게 됩니다.

- `WHERE id = ?`일 때 인덱스 사용
- 다른 컬럼 검색 시 선형 탐색 사용

이때 환경이 제각각이면 결과 해석이 어려워집니다.

물론 Docker를 쓴다고 성능이 완전히 똑같아지지는 않습니다.

그래도 적어도 아래는 맞출 수 있습니다.

- 같은 OS 계열
- 같은 패키지 버전
- 같은 빌드 명령
- 같은 실행 명령

즉, 성능 비교의 조건을 더 통제하기 쉬워집니다.

### 3. 팀 협업 중 "내 환경에서는 되는데?" 문제를 줄이기 위해

프로젝트가 커질수록 자주 나오는 문제가 있습니다.

```text
내 컴퓨터에서는 되는데?
```

이 말이 자주 나오면 디버깅 비용이 커집니다.

Docker를 쓰면 "이 컨테이너 안에서 실행했을 때"를 기준으로 대화를 맞출 수 있습니다.

즉, 문제 재현 기준이 통일됩니다.

### 4. 발표와 데모를 안정적으로 준비하기 위해

발표 직전에 가장 위험한 상황은 데모 환경이 달라지는 것입니다.

예를 들면 아래 같은 경우입니다.

- 발표용 PC에 필요한 도구가 없음
- 경로 설정이 다름
- 실행 스크립트가 OS별로 다름

Docker를 사용하면 "이 컨테이너에서 이 명령을 실행하면 된다"라는 형태로 데모 절차를 단순화할 수 있습니다.

## 현재 프로젝트에서 기대할 수 있는 Docker 사용 방식

이번 프로젝트에서는 보통 아래 흐름으로 사용할 가능성이 큽니다.

1. Dockerfile을 만든다.
2. C 빌드에 필요한 패키지를 설치한다.
3. 프로젝트 폴더를 컨테이너 작업 디렉터리로 연결한다.
4. 컨테이너 안에서 `make`, 테스트, 벤치마크를 실행한다.
5. 팀원 모두가 같은 방식으로 실행한다.

즉, Docker는 애플리케이션 기능 그 자체가 아니라,

**프로젝트를 안정적으로 개발, 테스트, 실행하기 위한 공통 기반 환경**

역할을 합니다.

## 이 프로젝트와 특히 잘 맞는 이유

현재 7주차 과제는 단순한 기능 추가가 아닙니다.

기존 SQL 처리기에 아래 요소를 붙여야 합니다.

- ID 자동 부여
- B+ 트리 자료구조
- 인덱스 기반 검색
- 대용량 데이터 생성
- 성능 측정 코드
- 테스트 자동화

이런 작업은 환경이 조금만 달라도 결과가 흔들릴 수 있습니다.

예를 들어 아래가 달라질 수 있습니다.

- 파일 경로 처리 방식
- 줄바꿈 처리 방식
- 빌드 옵션
- 시간 측정 방식
- 테스트 스크립트 동작 방식

Docker는 이런 흔들림을 줄여서 "기능 구현" 자체에 집중하게 도와줍니다.

## Docker를 쓴다고 모든 문제가 해결되지는 않는다

중요한 점도 있습니다.

Docker는 만능이 아닙니다.

아래 문제는 여전히 직접 해결해야 합니다.

- 코드 버그
- 잘못된 자료구조 설계
- B+ 트리 로직 오류
- 잘못된 성능 측정 방법
- 메모리 누수

즉, Docker는 **좋은 실행 환경을 만들어 주는 도구**이지, 코드 품질을 대신 보장해 주는 도구는 아닙니다.

## 초심자 관점에서 기억하면 좋은 문장

아래 문장만 기억해도 충분합니다.

- Docker는 실행 환경을 통째로 맞추기 위한 도구다.
- 이미지는 실행 환경 설계도다.
- 컨테이너는 그 설계도를 실제로 실행한 것이다.
- 현재 프로젝트에서는 팀원 모두가 같은 환경에서 빌드, 테스트, 벤치마크하기 위해 Docker가 필요하다.
- 7주차 과제에서는 특히 성능 테스트와 데모 재현성을 높이는 데 도움이 된다.

## 현재 프로젝트 기준 한 줄 정리

이번 7주차 SQL 처리기 프로젝트에서 Docker는

**B+ 트리 구현 자체를 위한 도구가 아니라, 그 구현을 모두가 같은 조건에서 안정적으로 개발하고 검증하기 위한 공통 실행 환경**입니다.
