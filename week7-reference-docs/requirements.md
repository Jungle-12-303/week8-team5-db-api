# SQL Processor 7주차 요구사항 정의서

## 1. 문서 목적

이 문서는 7주차 과제인 B+ 트리 인덱스 기능을 현재 SQL 처리기에 어떤 방식으로 추가할지 요구사항 수준에서 정의한다.

핵심 해석은 다음과 같다.

- 과제에서 말하는 `id`는 사용자 스키마에 명시되지 않아도 SQL 처리기 내부에서 자동 부여되는 숨은 PK다.
- 문서에서는 이 숨은 PK의 내부 구현명을 `__internal_id`로 통일한다.
- 사용자는 기존 6주차 방식대로 `INSERT`, `SELECT`, `SELECT ... WHERE`를 사용하되, 실행기는 `__internal_id`를 자동 관리해야 한다.
- `WHERE id = <value>`는 사용자 컬럼이 아니라 내부 `__internal_id` 기준 조회로 해석한다.

## 2. 목표

- 기존 SQL 처리기의 `INSERT`, `SELECT`, `SELECT ... WHERE` 흐름을 유지한다.
- 새 레코드에 대해 내부 `__internal_id`를 자동 부여한다.
- 내부 `__internal_id`를 B+ 트리에 인덱싱한다.
- `WHERE id = <value>`일 때 B+ 트리 인덱스를 사용한다.
- 대량 데이터 준비 후 인덱스 조회와 선형 조회 성능을 비교한다.

## 3. 범위

### 3.1 포함 범위

- `INSERT`
- `SELECT *`
- `SELECT column1, column2`
- `SELECT ... WHERE column = value`
- 숨은 내부 `__internal_id` 자동 생성
- `WHERE id = <number>` 인덱스 조회
- 메모리 기반 B+ 트리
- 대량 데이터 준비 및 조회 성능 측정

### 3.2 제외 범위

- `DELETE`
- `UPDATE`
- `JOIN`
- `ORDER BY`
- `GROUP BY`
- 복합 `WHERE`
- 범위 검색 최적화
- 동시성 제어
- 트랜잭션

## 4. 데이터 모델 해석

### 4.1 사용자 스키마

- 테이블 스키마는 `schema/<table>.meta`로 존재한다고 가정한다.
- 실제 레코드 파일은 `data/<table>.csv`로 존재한다고 가정한다.
- 각 테이블의 첫 줄은 사용자 컬럼 헤더다.
- 사용자 스키마에는 `id` 컬럼이 없어도 된다.

### 4.2 내부 `__internal_id`

- `__internal_id`는 사용자 컬럼이 아니라 SQL 처리기 내부에서 관리하는 숨은 키다.
- 새 레코드가 추가되면 실행기는 다음 `__internal_id`를 자동으로 부여해야 한다.
- `__internal_id`는 B+ 트리 인덱스의 key로 사용된다.
- 사용자는 SQL 문법에서 `WHERE id = <value>` 형태로 접근하지만, 실행기는 이를 `__internal_id` 조회로 해석한다.

### 4.3 명시적 `id` 컬럼 정책

- 7주차 목표 설계에서는 사용자 스키마의 명시적 `id` 컬럼 방식을 폐기한다.
- 사용자 스키마에 `id` 컬럼이 없어도 내부 `__internal_id` 자동 부여와 `WHERE id = ...` 조회가 가능해야 한다.
- 사용자 스키마에 `id`라는 컬럼명을 추가하는 것은 허용하지 않는 방향을 기본 정책으로 한다.
- 이유는 사용자 컬럼 `id`와 내부 예약 키 `__internal_id`가 같은 SQL 표면 문법 `WHERE id = ...`를 두고 충돌하기 때문이다.

### 4.3 append-only 전제

- 현재 프로젝트는 append 중심 흐름이며 `UPDATE`, `DELETE`가 없다.
- 따라서 CSV의 행 순서와 행 시작 오프셋은 삽입 이후 안정적으로 유지된다고 가정한다.
- 이 전제 아래에서 인덱스 재구성 시 CSV의 데이터 행 순서대로 `__internal_id`를 다시 1부터 부여할 수 있다.

## 5. 기능 요구사항

### 5.1 INSERT 요구사항

- 사용자는 기존 6주차 방식과 호환되는 `INSERT`를 실행할 수 있어야 한다.
- 사용자는 `id` 컬럼을 명시하지 않아도 된다.
- 실행기는 새 숨은 내부 `__internal_id`를 자동 생성해야 한다.
- 저장 전 새 레코드를 사용자 컬럼 기준 CSV 행으로 구성해야 한다.
- CSV 저장 직후 해당 `__internal_id`를 B+ 트리에 등록해야 한다.
- 저장 성공 시 인덱스와 CSV가 서로 일치해야 한다.
- CSV 저장 후 인덱스 등록에 실패하면 현재 실행은 오류를 반환해야 하며, 이후 인덱스는 CSV 기준 재구성을 통해 일관성을 회복할 수 있어야 한다.

예시:

```sql
INSERT INTO student (department, student_number, name, age)
VALUES ('컴퓨터공학과', '2024001', '김민수', 20);
```

### 5.2 SELECT 요구사항

- 기존 `SELECT *` 동작을 유지한다.
- 기존 특정 컬럼 조회 동작을 유지한다.
- `WHERE`가 없으면 기존처럼 전체 스캔으로 출력한다.
- `WHERE name = 'value'` 같은 일반 조건은 기존 선형 탐색을 유지한다.
- `WHERE id = <number>`이면 내부 `__internal_id` 기준 B+ 트리 인덱스를 사용한다.
- `WHERE id = <value>`에서 `<value>`는 정수여야 하고, 정수가 아니면 오류 처리해야 한다.
- `__internal_id`는 사용자 컬럼이 아니지만, 시연과 검증을 위해 `SELECT id`는 내부 `__internal_id`를 노출하는 방식으로 지원할 수 있다.
- `SELECT *`는 기존 의미를 유지하기 위해 사용자 스키마 컬럼만 출력하고, 숨은 내부 `__internal_id`는 자동 포함하지 않는다.

예시:

```sql
SELECT * FROM student WHERE id = 1000;
SELECT name, age FROM student WHERE department = '컴퓨터공학과';
```

### 5.3 인덱스 재구성 요구사항

- 프로그램 시작 시 특정 테이블 인덱스가 아직 메모리에 없으면 CSV를 읽어 인덱스를 재구성할 수 있어야 한다.
- 이미 메모리에 있는 인덱스는 같은 프로세스 안에서 재사용할 수 있어야 한다.
- REPL 또는 벤치마크 실행 중 반복 조회에서도 같은 인덱스를 재사용해야 한다.
- 인덱스 재구성 시 CSV 데이터 행을 처음부터 끝까지 읽으며 행 순서대로 `__internal_id`를 다시 부여해야 한다.
- 같은 CSV 내용이라면 같은 `id -> row_offset` 매핑이 재구성되어야 한다.

## 6. 성능 요구사항

- 대용량 레코드를 반복 가능하게 생성할 수 있는 입력 생성 경로 또는 데이터 생성 도구가 있어야 한다.
- 최소 1,000,000개 이상의 레코드를 준비할 수 있어야 한다.
- 이미 준비된 데이터셋을 기준으로 조회 성능만 따로 측정할 수 있어야 한다.
- `WHERE id = <value>` 조회 시간과 `WHERE other_column = <value>` 조회 시간을 비교할 수 있어야 한다.
- 같은 입력 파라미터 또는 같은 시드를 사용하면 같은 데이터셋을 반복 생성할 수 있어야 한다.

## 7. 오류 처리 요구사항

- 존재하지 않는 테이블 조회는 오류 처리해야 한다.
- 존재하지 않는 컬럼 조회는 오류 처리해야 한다.
- 잘못된 `WHERE` 문법은 오류 처리해야 한다.
- `WHERE id = <value>`에서 정수가 아닌 값을 오류 처리해야 한다.
- 인덱스 조회 시 해당 `id`가 없으면 빈 결과를 반환해야 한다.
- CSV 내용과 스키마 컬럼 수가 맞지 않으면 오류 처리해야 한다.
- 인덱스 재구성 중 CSV 행 파싱이 실패하면 오류 처리해야 한다.

## 8. 테스트 요구사항

### 8.1 단위 테스트

- B+ 트리 삽입
- B+ 트리 검색
- 노드 분할
- 중복 키 처리 정책
- 내부 `__internal_id` 자동 생성
- CSV 오프셋 계산
- `WHERE id = value` 분기 결정
- CSV 행 순서 기반 `__internal_id` 재부여 검증

### 8.2 기능 테스트

- `INSERT` 후 CSV 저장과 인덱스 등록이 모두 성공한다.
- `SELECT ... WHERE id = value`가 정확히 한 행을 반환한다.
- 일반 `WHERE`는 기존 방식대로 동작한다.
- 메모리 인덱스를 초기화한 뒤 다시 재구성해도 같은 `id` 조회 결과를 얻는다.
- CSV 저장 후 인덱스 등록 실패가 발생하면 다음 인덱스 접근 시 CSV 기준 재구성으로 복구할 수 있어야 한다.
- `id` 컬럼이 없는 테이블도 `__internal_id` 자동 부여와 `WHERE id = ...` 조회가 가능해야 한다.
- 데이터 준비와 인덱스 조회 성능 비교가 가능해야 한다.

## 9. 완료 기준

아래 조건을 모두 만족하면 7주차 필수 구현 완료로 본다.

- 기존 SQL 처리기의 실행 흐름을 유지한다.
- `INSERT` 시 숨은 내부 `__internal_id`가 자동 생성된다.
- 생성된 `__internal_id`가 B+ 트리에 등록된다.
- `WHERE id = <value>` 조회가 인덱스를 사용한다.
- 일반 컬럼 조회는 기존 선형 탐색으로 유지된다.
- 사용자 스키마의 명시적 `id` 컬럼 없이도 7주차 흐름에서 사용할 수 있다.
- 1,000,000건 이상의 데이터 준비 및 조회 성능 측정 경로가 존재한다.
- 단위 테스트와 기능 테스트가 존재한다.
