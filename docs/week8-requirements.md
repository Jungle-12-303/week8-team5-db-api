# SQL Processor 8주차 요구사항 정의서

## 1. 문서 목적

이 문서는 8주차 수요 코딩회 과제인 `미니 DBMS - API 서버` 구현 범위를 요구사항 수준에서 정의한다.

본 문서의 1차 기준은 공식 공지 문서인 8주차 수요 코딩회 공지사항이다.

## 2. 공식 요구사항 원문 기준 해석

공식 공지에서 직접 확인되는 핵심 요구는 다음과 같다.

- 이번 주 과제는 `미니 DBMS - API 서버` 구현이다.
- 구현한 API를 통해 외부 클라이언트가 DBMS 기능을 사용할 수 있어야 한다.
- 스레드 풀을 구성하고, 요청이 들어올 때마다 스레드를 할당하여 SQL 요청을 병렬로 처리해야 한다.
- 이전 차수에서 구현한 SQL 처리기와 B+ 트리 인덱스를 그대로 활용하여 내부 DB 엔진을 구성해야 한다.
- 구현 언어는 C 언어다.
- 단위 테스트와 API 서버 기능 테스트를 수행해야 한다.
- 엣지 케이스를 최대한 고려해야 한다.
- README.md를 기준으로 결과물을 설명할 수 있어야 한다.

즉 8주차의 본질은 `새로운 DB 엔진을 처음부터 다시 만드는 것`이 아니라, `기존 SQL 처리기와 B+ 트리 인덱스를 내부 엔진으로 재사용하면서 외부 API 서버와 병렬 처리 구조를 추가하는 것`이다.

## 3. 목표

- 기존 SQL 처리기와 B+ 트리 인덱스를 내부 DB 엔진으로 재사용한다.
- 외부 클라이언트가 네트워크 API를 통해 DBMS 기능을 사용할 수 있게 한다.
- 스레드 풀 기반 요청 처리 구조를 제공한다.
- 동시에 들어오는 SQL 요청을 병렬로 처리할 수 있게 한다.
- 병렬 처리 상황에서도 데이터 정합성과 서버 안정성을 유지한다.
- 단위 테스트와 API 기능 테스트를 통해 구현 결과를 검증한다.
- README.md만으로 프로젝트 목적, 구조, 실행 방법, 검증 방법을 설명할 수 있게 한다.

## 4. 범위

### 4.1 포함 범위

- C 언어 기반 API 서버
- 외부 클라이언트 요청 수신
- 요청을 내부 SQL 처리기 실행으로 연결하는 서버 계층
- 이전 차수 SQL 처리기 재사용
- 이전 차수 B+ 트리 인덱스 재사용
- 스레드 풀 기반 병렬 요청 처리
- SQL 요청 실행 결과를 클라이언트에 응답하는 기능
- 단위 테스트
- API 서버 기능 테스트
- 엣지 케이스 검증

### 4.2 제외 범위

공식 공지에서 아래 항목은 필수로 요구하지 않는다.

- 새로운 DB 저장 엔진으로의 전면 교체
- 기존 SQL 문법 범위를 넘어서는 대규모 확장
- 분산 처리
- 인증/인가
- 트랜잭션, MVCC, 고급 격리 수준
- 고급 REST 리소스 모델 설계
- 운영 환경 배포 자동화

위 항목은 추가 구현 요소로는 가능하지만, 8주차 필수 완료 조건으로 간주하지 않는다.

## 5. 시스템 요구사항

### 5.1 내부 DB 엔진 재사용

- 서버는 이전 차수에서 구현한 SQL 처리기를 내부 DB 엔진으로 사용해야 한다.
- 서버는 이전 차수에서 구현한 B+ 트리 인덱스를 내부 조회 최적화 구조로 유지해야 한다.
- 기존 SQL 처리기 계층 구조와 책임 분리를 가능한 한 유지해야 한다.
- API 서버 추가를 이유로 컨트롤러 성격의 계층이 직접 저장소에 접근하거나, 기존 계층 책임을 무너뜨리는 방식은 지양해야 한다.

### 5.2 외부 API 제공

- 외부 클라이언트는 API 요청을 통해 내부 DB 엔진 기능을 사용할 수 있어야 한다.
- API 서버는 클라이언트 요청을 받아 적절한 SQL 실행 흐름으로 연결해야 한다.
- API 서버는 실행 성공, 사용자 입력 오류, 내부 실패를 구분 가능한 응답으로 반환해야 한다.
- 요청 및 응답 형식은 팀이 선택할 수 있으나, README와 테스트에서 재현 가능하게 명시해야 한다.
- 서버 실행에 필요한 설정값과 데이터 경로 주입 방식은 문서화되어야 한다.
- 요청 크기 제한, SQL 길이 제한, 지원하지 않는 HTTP 기능에 대한 처리 정책은 문서화되어야 한다.

### 5.3 SQL 요청 처리

- API 서버는 SQL 실행 요청을 받아야 한다.
- SQL 요청은 기존 SQL 처리기가 지원하는 범위 안에서 처리해야 한다.
- 기존 엔진이 지원하지 않는 SQL은 명확한 오류로 처리해야 한다.
- 기존 B+ 트리 인덱스를 활용하는 조회 경로는 API 서버 환경에서도 유지되어야 한다.

## 6. 병렬 처리 요구사항

### 6.1 스레드 풀

- 서버는 스레드 풀을 구성해야 한다.
- 요청이 들어올 때마다 작업을 스레드 풀의 작업자 스레드에 할당할 수 있어야 한다.
- 요청 처리를 위해 매 요청마다 새 스레드를 만드는 구조를 기본 방식으로 삼아서는 안 된다.

### 6.2 병렬 SQL 처리

- 서버는 동시에 들어오는 SQL 요청을 병렬로 처리할 수 있어야 한다.
- 병렬 처리 과정에서 서버가 비정상 종료되거나 공유 상태가 쉽게 손상되지 않아야 한다.
- 공유 자원 접근에는 동시성 제어가 필요하다.

### 6.3 동시성 안전성

- 기존 SQL 처리기와 인덱스가 원래 단일 실행 흐름을 전제로 했다면, API 서버는 이를 병렬 실행 환경에서 안전하게 보호해야 한다.
- 데이터 파일, 인덱스 구조, 전역 레지스트리, 자동 증가 ID 같은 공유 상태는 경쟁 조건 없이 다뤄져야 한다.
- 읽기와 쓰기가 동시에 발생할 때도 결과의 정합성을 최대한 보장해야 한다.
- 락 범위와 직렬화 정책은 구현 선택 사항이지만, 어떤 자원을 어떤 기준으로 보호하는지 README와 문서에서 설명 가능해야 한다.

## 7. 품질 요구사항

### 7.1 단위 테스트

- 주요 함수 또는 모듈은 단위 테스트로 검증해야 한다.
- 특히 요청 처리 보조 로직, 작업 큐, 스레드 풀, 동시성 보호 로직, 엔진 연결부는 단위 테스트 대상에 포함되어야 한다.

### 7.2 API 기능 테스트

- API 서버를 실제로 실행한 상태에서 기능 테스트를 수행해야 한다.
- 정상 요청과 비정상 요청 모두 테스트해야 한다.
- SQL 실행 성공, SQL 오류, 잘못된 요청 형식, 존재하지 않는 경로 또는 기능 제한 케이스를 점검해야 한다.

### 7.3 엣지 케이스

- 빈 요청 또는 잘못된 요청을 안전하게 처리해야 한다.
- 잘못된 SQL, 지원하지 않는 SQL, 잘못된 파라미터를 안전하게 처리해야 한다.
- 과도하게 큰 요청 본문, 과도하게 긴 SQL, 잘못된 헤더, 지원하지 않는 전송 방식도 안전하게 처리해야 한다.
- 잘못된 서버 시작 옵션이나 존재하지 않는 데이터 경로도 명확한 시작 실패로 처리해야 한다.
- 동시 요청 상황에서 race condition, deadlock, unlock 누락, 중복 ID 생성, 인덱스와 파일 불일치 같은 위험을 점검해야 한다.
- 서버 종료 또는 오류 경로에서도 자원 정리가 누락되지 않아야 한다.

- 그러나 선택한 방식은 테스트 가능하고 README에서 설명 가능해야 한다.
